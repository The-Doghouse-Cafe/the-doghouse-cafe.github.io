# Dog House Cafe Agent Notes

## Repo Structure

- The site is an Astro app rooted at this repository.
- Keep `README.md` at the repository root.
- Project docs belong under `docs/`.
- Use `docs/authors.md` as the guide for people and author data.
- Root pages live at `src/pages/index.astro`, `src/pages/about/index.astro`, and `src/pages/people.astro`.
- Blog pages live under `src/pages/blog/`.
- People profile pages are generated by `src/pages/people/[slug].astro`.
- Shared static styling is authored in `src/styles/base.css` and served from `public/styles/base.css`.
- Theme variants are authored in separate CSS files under `src/styles/` and served from `public/styles/`.
- Theme switching behavior lives in `public/scripts/theme.js`.

## People And Authors

- Internal people records live in `src/content/people/`.
- `src/content/people/_template.md` is the canonical base template for internal people records.
- Blog-only or external author fallback data lives in `src/blog-content/authors/`.
- People data is normalized through `src/lib/people.ts`.
- When a matching slug or name exists, `src/content/people/` overrides matching records from `src/blog-content/authors/`.
- Public people routes are `/people` and `/people/[slug]`.
- `_template.md` files must not render as real people.
- Files with `draft: true` should not render publicly.
- Prefer adding or updating people data in `src/content/people/` instead of hard-coding people in components.
- Blog author avatars and people images resolve from `public/assets/people/`; run `.tools/sync-people-images.sh` when copying source image assets into the people assets folder.

## Blog Work

- Use the root Astro project for blog-related work.
- Astro config: `astro.config.mjs`
- Astro base layout: `src/layouts/BaseLayout.astro`
- Blog post layout: `src/layouts/PostLayout.astro`
- Blog post content: `src/blog-content/posts/`
- Blog author fallback content: `src/blog-content/authors/`
- Astro source pages: `src/pages/`
- Package scripts are defined in `package.json`.
- Run commands from the repository root, such as `npm run dev`, `npm run build`, or `npm run preview`.
- Keep blog UI, routes, content, and Astro-specific assets inside `src/`.
- Astro pages should use `src/layouts/BaseLayout.astro` so theme styling and shared page chrome stay aligned.
- Blog posts should be Markdown files with frontmatter matching `src/content.config.ts`.
- Post frontmatter should include an `author` object with `name`, plus optional `role`, `avatar`, and `linkedin` fields.
- Author LinkedIn links must be HTTPS LinkedIn URLs.

## Theme System

- `src/styles/base.css` contains shared layout, components, page styles, and theme switcher styling.
- `src/styles/ember.css` is the default warm dark theme.
- `src/styles/ocean.css` is the darker ocean-inspired theme.
- When changing colors, shadows, gradients, or role palette values, update the theme files rather than hard-coding visual values in `base.css`.
- Preserve the token contract between `base.css` and each theme file.
- After changing files in `src/styles/`, run `.tools/sync-styles.sh` so `public/styles/` is updated.

Each theme should define the full token set used by `base.css`, including:

- `--bg`
- `--panel`
- `--text`
- `--muted`
- `--accent`
- `--accent-soft`
- `--border`
- `--body-bg`
- `--logo-shadow`
- `--button-hover-border`
- `--button-primary-text`
- `--profile-bg-start`
- `--profile-bg-mid`
- `--profile-bg-deep`
- `--founder-bg-end`
- `--founder-glow-color`
- `--dev-bg-end`
- `--dev-glow-color`
- `--author-bg-end`
- `--author-glow-color`
- `--card-overlay`
- `--footer-border`
- `--footer-text`
- `--footer-link-hover`

People cards use these role classes:

- `people-card founder`
- `people-card dev`
- `people-card author`

The role classes set local `--bg-end` and `--glow-color` values, which drive the profile card gradient and avatar ring.

## Theme Switcher

Pages that support themes should include:

- A stylesheet link with `data-theme-link`, defaulting to `/styles/ember.css`.
- A `.theme-switcher` control with a `[data-theme-select]` select.
- `/scripts/theme.js` near the end of the body.

The switcher currently supports `ember` and `ocean`, persists the selected theme in `localStorage` under `doghouse-theme`, and swaps the active stylesheet to `/styles/{theme}.css`.

When adding a new theme:

1. Create `src/styles/{theme}.css`.
2. Import `./base.css`.
3. Define the full token set.
4. Run `.tools/sync-styles.sh` so the theme is available from `public/styles/`.
5. Add the theme name to `themes` in `public/scripts/theme.js`.
6. Add an option to each `[data-theme-select]` control.

## Development Workflow

- Main development flow is feature branch to `dev` to `main`.
- Netlify deploy previews handle PR previews.
- Netlify builds should use `npm ci && npm run build` and publish `dist`.
- Use `npm run build` as the baseline verification before handing off changes.

## Editing Guidance

- Prefer extending existing classes, tokens, and helper APIs over adding one-off styles or data loaders.
- Use named theme files such as `ember.css` and `ocean.css`; do not add a generic `dark.css` shim.
- If a new root page is added, include the theme switcher pattern so theme selection remains consistent across the static site.
- Keep generated or served copies, such as `public/styles/base.css`, in sync with the authored source when the repo workflow requires it.

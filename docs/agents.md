# Contributor Guide

This guide summarizes the project conventions contributors should follow when
working on the Dog House Cafe site.

## Project Structure

- This repository is an Astro site.
- Keep `README.md` at the repository root.
- Keep contributor-facing documentation in `docs/`.
- Root pages live in:
  - `src/pages/index.astro`
  - `src/pages/about/index.astro`
  - `src/pages/people.astro`
- Blog pages live under `src/pages/blog/`.
- People profile pages are generated by `src/pages/people/[slug].astro`.
- Shared layouts live in `src/layouts/`, especially `src/layouts/BaseLayout.astro`.
- Reusable UI components live in `src/components/`.
- Blog posts live in `src/blog-content/posts/`.
- People and author data is described in [docs/authors.md](./authors.md).

## Development Workflow

Run commands from the repository root.

```sh
npm run dev
npm run build
npm run preview
```

Use `npm run build` as the baseline verification before handing off or opening a
pull request.

Astro pages should use `src/layouts/BaseLayout.astro` so shared page chrome,
theme styles, and navigation stay consistent.

## Branch/PR Workflow

- Main development flow is feature branch to `dev` to `main`.
- Open pull requests against `dev` unless the change is explicitly intended for
  `main`.
- Netlify deploy previews are used for PR review.
- Netlify builds should run `npm ci && npm run build` and publish `dist`.

## People/Authors Data Model

The site has a unified people model for team members, contributors, and blog
authors. See [docs/authors.md](./authors.md) for field-level details and
examples.

In short:

- Internal people records live in `src/content/people/`.
- Use `src/content/people/_template.md` as the base for new internal people.
- Blog-only or external fallback author data lives in `src/blog-content/authors/`.
- People data is normalized through `src/lib/people.ts`.
- If a matching slug or normalized name exists, `src/content/people/` overrides
  fallback data from `src/blog-content/authors/`.
- Public people routes are `/people` and `/people/[slug]`.
- `_template.md` files must not render as real people.
- Records with `draft: true` should not render publicly.
- Prefer adding or updating people data instead of hard-coding people in
  components.
- Blog author avatars and people images resolve from `public/assets/people/`.
  Run `.tools/sync-people-images.sh` after copying source image assets into the
  people assets folder.

## Build/Deploy Notes

- Package scripts are defined in `package.json`.
- The production build output is `dist`.
- Treat `npm run build` as the required local check for layout, content, and
  route generation changes.
- Keep served/generated copies in sync with authored files when the workflow
  requires it.

## Styling/Layout Conventions

- Shared static styling belongs in the main base stylesheet.
- Theme-specific colors, shadows, gradients, and role palette values belong in
  named theme files such as `ember.css` and `ocean.css`.
- Preserve the token contract between the base stylesheet and theme files.
- Do not add a generic `dark.css` shim.
- If editing authored theme files, run `.tools/sync-styles.sh` so
  `public/styles/` is updated.
- Extend existing classes, tokens, and helper APIs before adding one-off styles.
- People cards use role classes:
  - `people-card founder`
  - `people-card dev`
  - `people-card author`
- Those role classes drive profile card gradients and avatar ring colors through
  local `--bg-end` and `--glow-color` values.

## Theme Switcher

Pages that support themes should include:

- A stylesheet link with `data-theme-link`, defaulting to `/styles/ember.css`.
- A `.theme-switcher` control with a `[data-theme-select]` select.
- The theme script near the end of the body.

The current themes are `ember` and `ocean`. The selected theme is stored in
`localStorage` under `doghouse-theme`.

When adding a theme:

1. Create a named theme stylesheet.
2. Import the shared base stylesheet.
3. Define the full theme token set.
4. Sync the theme into `public/styles/`.
5. Add the theme to the theme script.
6. Add an option to each theme select control.

## What Not To Change Casually

- Do not remove or rewrite `AGENTS.md`; it contains automation-facing guidance.
- Do not hard-code people or author records in components.
- Do not let `_template.md` or `draft: true` records render publicly.
- Do not change merge behavior between `src/content/people/` and
  `src/blog-content/authors/` without updating [docs/authors.md](./authors.md).
- Do not hard-code theme colors in shared layout styles when a token belongs in
  the theme files.
- Do not rename public routes casually; `/people`, `/people/[slug]`, and blog
  routes are part of the site structure.
- Do not make broad layout refactors in the same change as content updates.
